# WORKBENCH — 工作台(每日生成)规范

只在处理"工作台"相关任务时读这份文件(用户点名提到工作台、要改 `工作台.md`、或跑 `daily-workbench.yml`/`weekly-gc.yml` 里涉及工作台的步骤)——**根规则文件不再内联这节内容,平时聊天不需要加载这份文件**。

根目录 `工作台.md` 是记忆库的出口:让旧记忆主动找上门,核心是提取练习而不是重读。页面固定为 **☀️ 每日回路**(`##` 标题,AI 板块)。每天由定时任务重新生成 AI 板块,其余部分(`##` 标题、所有 dataviewjs 区块——横幅、信息条、今日内容折叠壳、胶囊标签栏(笔记/概念/本周新增/最近编辑四颗点开在页面内展开面板,双链胶囊直接跳原生图谱不展开面板,➕ 新建在最右)、🎲 随机重逢卡)是固定结构或实时渲染,**生成时原样保留,永远不要改动**。

**笔记面板**点分类卡打开的是 `meta/笔记数据库.base`(Obsidian 原生 Bases,每次点开笔记面板都会重新生成,视图按 `notes/` 当前的子文件夹实时同步)——这是生成产物,不是用户笔记,不要手动编辑,也不要把它当成笔记内容去索引/复核。格式保持现有 callout 语法(`[!today]` `[!draw]` `[!seed]` `[!ask]` `[!note]` `[!write]` `[!clash]`);frontmatter 保留 `cssclasses: workbench`,但每次**必须把 `generated:` 更新为当天日期**——横幅心跳报警靠它判断每日任务是否还活着(超 48 小时未更新显红),这是"系统静默挂掉"唯一的可见信号。云端日更由 workflow 前置 sed 步骤机械写入,AI 原样保留即可;只有人工/本地更新工作台时才需要手动改这个字段。


**视觉基调**:由横幅画作在客户端用 canvas 取主色调自动算出(色相分桶投票,不用平均色——平均会把所有颜色互相抵消成灰)、写进 CSS 变量 `--wb-accent`,CSS 里再用 OKLCH 相对颜色语法派生锁定亮度的文字色/柔光底。

**🖼 每日画作**(横幅背景):每天生成时用执行器可用的网页搜索工具挑一幅著名画作(Codex 自动化使用内置 web search),优先 Wikimedia Commons 上 `upload.wikimedia.org` 开头的直链图片(几百 KB 到 2MB 的中等分辨率版本,不要几十 MB 的原图),写入 frontmatter 三个字段:`painting-url`、`painting-title`(画名·作者)、`painting-intro`(一句话简介)。**链接必须是当次搜索/抓取真实确认可访问的,查不到就保留前一天的画,不编造 URL**。选画不许与近半年重复(查重命令:`git log --since='180 days ago' --format= -p -- 工作台.md | grep '^+painting-title'`,输出仅每天一行画名,省 token 且永久封顶;不要跑裸的 `git log -p`),并在时代/地域/风格上轮换,不要连续几天同一风格。

每日生成三个板块,都在 ☀️ 每日回路的 `[!today]+` 折叠壳里(标题留空):

- **🎴 今日抽卡**:从 `notes/` 挑"最久没被抽过"的(查工作台历史 git log 去重复),出 1-3 道回忆提示——目的是点开笔记前先在脑子里过一遍核心内容。
- **🌱 今日延伸**:挑一个概念页(优先最近被链到的、或延伸区还空着的),补一条外部知识,标 [事实]/[推论]/[观点]、注明未联网核查;同步写进该概念页延伸区。**引用某篇笔记的录入时间时用实际日期。
- **❓ 今天的问题**:每天出**两道**独立问题(问题1、问题2),优先从概念页开放问题、或笔记间张力里出;**问题形式不限**——可以是指向用户自身经验的具体提问,也可以是纯开放式的观点/推演题,不强制绑定个人经历,两道题尽量来自不同概念/笔记,避免同质。**页面顺序固定为两张问题卡紧挨着,不能打乱或在中间插别的块**——靠这个顺序 + 页面级卡片网格的 flex 换行,两张问题卡自动横排成一行:
  1. `[!ask]` 问题1 → `[!ask]` 问题2:标题写 `❓ 问题1`/`❓ 问题2`,内容第一段是问题正文;**紧接着先嵌套** `[!write]` 子 callout(嵌套写法:问题的引用块内再用 `> >` 包一层),标题固定 `✍️ 你的回答`,内容留 2 行空的嵌套引用行(单独一个 `> >`)给用户手写,不预设任何文字
  2. **再嵌套**一个折叠的 `[!note]-` 子 callout(同样 `> >` 写法),标题固定 `💬 我的看法`,内容是 AI 对应问题的倾向性判断,标注 [推论]/[观点],不假装客观,折叠是 honor system,展开与否交给用户自觉。**2026-07-07 二次调整:回答区要排在"我的看法"前面**——用户要求写自己的答案时不能先看到 AI 的看法,避免被提前锚定,所以顺序固定"问题正文 → 你的回答 → 我的看法(折叠)",不能颠倒。**2026-07-07 起回答区并进对应问题卡内部,不再是独立顶层卡片**(独立空卡看着像 UI 没加载完,跟问题的配对关系也读不出来),"输入井"凹陷样式由 CSS 管,生成时只管结构

**生成前必做**:读上一版每张问题卡里嵌套 `[!write]` 的回答。**按文档顺序对应**:第一个 `[!write]` 是问题1的回答,第二个是问题2的回答(2026-07-07 起嵌在问题卡内部,顺序对应关系不变)。**分别独立判断**,哪个 `[!write]` 里有内容就处理哪道:

1. **归档(无条件)**:问题原文+回答原文追加到 `notes/问答/YYYY-MM 工作台问答.md`(按月一个文件,`type: note`、`checkable: false`)。**先归档再覆盖,顺序不能反**
2. **沉淀(有价值时)**:回答里的新判断/新张力/值得连的概念,留痕到相关概念页或笔记
3. **碰撞(有回答时必须生成)**:在新工作台 🎴 抽卡**之前**,每道被回答的问题各插入一个 `[!clash]-` callout(**默认折叠,不要改回展开**;两道都答了就两个,按问题1/2顺序排),标题 `🥊 碰撞 · 问题1 回顾`/`🥊 碰撞 · 问题2 回顾`,内容四段:「**问题：**」(一句话)、「**你的回答：**」(原文摘要,不改意)、「**我的看法：**」(从上期折叠 callout 里搬来,完整展示)、「**碰撞：**」——在这段里做真正的分析:指出两者的核心分歧在哪、谁的角度更贴近实际经验、哪里 AI 在偷懒或过度抽象、哪里用户可能在自我美化或回避——**不要和稀泥,不要都说有道理,要有立场**
