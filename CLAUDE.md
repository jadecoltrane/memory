# CLAUDE.md — AI 协作与记忆库规则

`jadecoltane/memory` 就是记忆库本身:数据层纯 Markdown + Git(可迁移,任何 AI 都能接入),使用层 Obsidian 插件随便用(Dataview、Git 等)。当前只接入 Claude。

**隐私边界**:含健康/心理/生活隐私,仓库必须保持 GitHub **private**,不得转公开、不得对外发布记忆内容,引用仅限用户本人会话(教训见 [[pitfalls/记忆库仓库曾默认公开导致隐私内容对外暴露]])。

**兜底优先**:可靠性相关的东西(失败告警、机器校验、边界情况、备份)必须**预先**设计,不等出事再补;懒加载只允许用在内容组织(概念页第二次命中才建、SUMMARY 凑够 2 条才起条目),不适用于安全网。

**方案设计优先省 token**:几种做法效果相当时选更省 token(读/维护成本更低)的,但不能拿效果去换(见 [[decisions/自动化与文档设计优先选省token不影响效果的轻量方案]])——本文件和下面提到的三个自动化 prompt 都按这个标准写,发现冗余可以精简,但不能牺牲可执行性。

**三个自动化,不需要 AI 手动补**:
- `rebuild-index.yml`:推送到 main 且改了记忆文件就自动重建 `index.md`(AI 仍可手动跑 `scripts/build_memory_index.py` 提前核对)
- `weekly-gc.yml`:每周一 09:00(北京时间)云端跑「Memory GC」一节的步骤 + 指针巡检(需要 `REPO_PATROL_PAT`,没配就跳过巡检)
- `lint.yml`:每次推送跑 `scripts/lint_memory.py` 查断链和 frontmatter 完整性,失败邮件通知(幽灵概念是设计允许的,不算断链)。AI 写完记忆建议本地先跑一遍

## 开工时(必做)

1. 读根目录 `index.md`(索引,每条记忆一行)
2. 只按需打开与当前任务相关的记忆文件,不要全量阅读

## 收工时(必做)

判断本次会话是否产生了值得沉淀的记忆。**该往哪个目录写、写不写,查根目录 `RESOLVER.md`**(判断逻辑单独维护在那,这里不重复)。

写完后运行 `python3 scripts/build_memory_index.py` 重新生成索引;若新记忆和已有记忆同属一个主题(2 条以上),同步维护 `SUMMARY.md`(见「结论速览」一节);记忆、索引、速览一起提交。

**写入必须告知**:任何时候写入或修改了记忆(自动沉淀或用户触发都一样),必须在给用户的回复末尾明确说出来,格式如:「🧠 已写入记忆:`decisions/xxx`」。禁止悄悄写入。

## 用户画像(profile/)——记录"用户是谁"

记忆库优先记录用户,而不是记录它自己。`profile/` 按主题一页,按需新增(职业、教育、人际、健康、价值观……),**不需要触发词,AI 主动维护**,写入照样按「写入必须告知」执行。

- **客观事实**(职业、城市、经历、家庭等)→ 说一次就记,不用等第二次
- **AI 归纳的软特质**(偏好、风格、性格倾向)→ 用户明确表达过,或不同场合重复出现 ≥2 次才写,防止过度解读
- 每条特质附一句依据(哪次对话、哪篇笔记观察到)和日期;特质变了就地更新并注明,不另开新文件
- **不做双链、不进图谱**:profile 正文不挂 `[[concepts/xxx]]`,引用其他记忆用纯文本路径。图谱只属于用户的知识(notes + concepts),本地 Obsidian 图谱过滤器同时排除 profile/decisions/pitfalls/insights/questions 和根目录基础设施文件
- frontmatter:`type: profile`、`updated: YYYY-MM-DD`
- 开工时跟 index.md 一起顺手扫一眼
- 用户否决权适用:说哪条不对,当场改,不辩护

## 我的收藏(notes/)——文章、知识点、用户自己写的东西

`notes/` 只放**文章、知识点、用户自己的想法/创作/经历**——不放 AI 协作规则、对 AI 行为的约束(那些归 `decisions/` 或 `profile/`,判断标准见 `RESOLVER.md`)。**永远不要自动写入,只响应用户的明确触发或用户自己手写的内容。**

- **触发词命中时**(「记下来」「存一下」「收藏这条」「写进记忆库」「remember this」),先按 `RESOLVER.md` 判断内容性质,只有文章/知识点/自己写的东西才走下面流程:
  1. 提炼成一条笔记(不是全文粘贴)
  2. frontmatter 用 `type: note`,加一行 `source:`;可自由建子文件夹分类,索引脚本递归收录
  3. 写入本仓库、重跑索引、提交推送到 main。**"记下来"永远指写进这个 git 仓库**,不是会话内的短期记忆或任何 AI 产品自带的记忆功能
  4. 当前会话没有本仓库时先获取(远程会话用 add_repo 添加 jadecoltane/memory;本地会话直接操作用户磁盘上的 clone),实在无法访问时把写好的内容整段输出给用户手动保存
- **先判断是否"可查证"**:
  - **事实性/可查证**(数据、健康、历史、科学结论、他人转述的客观陈述等):走下面的 verified + 复核流程
  - **主观性/不可查证**(个人想法、感受、散文、创作、经历叙述、审美偏好):frontmatter 加 `checkable: false`,**不设 verified 字段,永远不进复核队列**——这类内容没有"对不对",原样留存即可
- **verified 标记**(仅 `checkable` 内容需要):内容来自用户自己亲历/验证过的判断 → `verified: true`;转述自其他 AI、网络、他人、未经交叉核实 → `verified: false`。拿不准时默认 `false`
- **GC 规则**:notes 只做去重合并,**不自动降级、不自动删除**——删除权只在用户

### notes 的复核(反茧房,防止未经查证的内容被当真理囤积)

**仅对 `checkable` 内容生效**;`checkable: false` 的笔记跳过本节全部流程。

**复核节奏**(按上次结果动态调整,不是查过一次就永远每月重查):

- 从未复核过的 `verified: false` note:30 天内必查
- 上次结果是「支持」:间隔延长到 180 天再查
- 上次结果是「存疑」或「找不到依据」:下次 GC(约一周后)继续查,直到用户裁决或结论转为「支持」

判定到期后:

1. **只用工具实际检索到的结果,禁止凭训练记忆编造引用**——引用链接必须来自 WebSearch/WebFetch 当次返回的真实 URL,没查到就写"未找到可靠来源"
2. 判断结论:支持 / 有争议或部分不准 / 找不到依据
3. **绝不修改或删除原文**——在文件末尾追加 `## 复核记录` 区块(没有则新建),格式:`- YYYY-MM-DD:结论,简述依据 + 真实来源链接,并注明下次复核大致时间`
4. 「存疑」或「找不到依据」的条目列入当周摘要提醒用户,由用户决定是否修改、标记 `verified: true` 或删除
5. 「支持」的静默追加记录即可
6. `verified: true` 的 note 不做真实性复核,仍适用「180 天未引用」使用度检查

### 让 notes 活起来:概念关联与反思

不只是攒文件,笔记之间要长出联想、张力、甚至矛盾——这是双链图谱存在的意义。

- **每次新写一条 note 时**(不管是响应触发写的,还是发现用户手写、还没建链接的笔记):扫一眼 `notes/` 里主题相近的旧笔记(用 index.md 定位候选),找**呼应或冲突**的双链上,双向都加
- **用户在 Obsidian 手写、推送到 GitHub 的笔记**(含错放在 notes/ 之外的散装文件、内容混了几个主题的):每周 GC(详细扫描算法见「Memory GC」4a)或用户说「帮我梳理一下 notes」时处理——补分类、frontmatter、双链,顺手做一遍"张力扫描"(笔记间/新旧笔记间矛盾,发现了原样列进摘要提问题,不给结论,不替用户调和)。**这一步允许直接改写原文**(拆分、改名、移动),依据见 [[decisions/整理手写笔记允许改写原文而不仅是追加复核记录]]
- **推送是前提**:在 Obsidian 里想怎么写都行,唯一要做的是让 Obsidian Git 插件推送到 GitHub——没推送等于我不知道这条笔记存在
- 找不到有价值的关联或张力时不要硬凑,宁可什么都不说

### 图片(截图等)

用户发图片(如小红书截图)并说「记下来」时:

1. 原图保存到 `notes/assets/YYYY-MM-DD-简短描述.png`(或原格式,不要转码压缩)
2. 对应 note 里用 `![说明](assets/文件名)` 嵌入原图,**并且必须额外写一段文字提炼**——图片本身无法被索引和 grep,未来 AI 靠这段文字才知道这条记忆在讲什么
3. 单张截图正常几百 KB 到几 MB,GitHub 单文件上限 100MB,无需顾虑;但不要把长视频或高清大图仓库当网盘用

## 概念层(concepts/)——图谱的枢纽节点

笔记是文档,概念才是思考的单位。`concepts/` 每页一个原子概念,让图谱长出概念连接而不是文档互相引用。页面结构四段:一句话定义(用用户自己的话)、出现在(链回笔记)、相关概念(概念互链)、延伸+开放问题(AI 补充)。

**双链规则**:笔记正文双链优先指向概念页(`[[concepts/某概念]]`),**笔记之间不直接互链**——例外:两篇确实是同一论证的上下篇时可以直链。

**同义词用 aliases 收编,不开新节点**:概念页 frontmatter 加 `aliases: [同义表述1, ...]`;写双链前先扫一眼 index 的概念列表和已有 aliases。GC 时发现近义幽灵概念,合并进实体页 aliases 并改写旧链接。

**懒加载**(控制 token 成本):

1. **录入时只做最小动作**:概念出现处写 `[[concepts/xxx]]`,概念页不存在也照写(图谱 `hideUnresolved: false`,幽灵节点零成本)
2. **第二次命中才建页**:被第二篇笔记链到时才创建实体页,只写「定义+出现在」,延伸留空
3. **延伸和开放问题只在两个时机写**:用户主动要求展开时;每周 GC 挑 1-2 个本周新命中的概念补。**绝不在录入时顺手写延伸**

frontmatter 用 `type: concept`;延伸区每条标注 [事实]/[推论]/[观点],凭训练知识写的注明"未联网核查",禁止编造引用链接。

## 工作台(每日生成)

根目录 `工作台.md` 是记忆库的出口:让旧记忆主动找上门,核心是提取练习而不是重读。每天由定时任务重新生成三个 AI 板块,其余部分(dataviewjs 横幅统计卡、📥 最近输入、👻 幽灵概念)是实时渲染,**生成时原样保留,永远不要改动**。格式保持现有 callout 语法(`[!draw]` `[!seed]` `[!ask]`);frontmatter 保留 `cssclasses: workbench`,但每次**必须把 `generated:` 更新为当天日期**——横幅心跳报警靠它判断每日任务是否还活着(超 48 小时未更新显红),这是"系统静默挂掉"唯一的可见信号。

每日生成三个板块:

- **🎴 今日抽卡**:从 `notes/` 挑"最久没被抽过"的(查工作台历史 git log 去重复),出 1-3 道自测题——考核心结构不考细节,不剧透答案。**抽几篇随体量扩容**:默认 1 篇,notes 超 100 篇后 2 篇、超 250 篇后 3 篇
- **🌱 今日延伸**:挑一个概念页(优先最近被链到的、或延伸区还空着的),补一条外部知识,标 [事实]/[推论]/[观点]、注明未联网核查;同步写进该概念页延伸区
- **❓ 今天的问题**:优先从概念页开放问题、或笔记间张力里出;问具体的、指向用户自身经验的问题

**生成前必做**:读上一版 ❓ 区块里用户的回答(✍️ 虚线框内)。有回答时:

1. **归档(无条件)**:问题原文+回答原文追加到 `notes/问答/YYYY-MM 工作台问答.md`(按月一个文件,`type: note`、`checkable: false`)。**先归档再覆盖,顺序不能反**
2. **沉淀(有价值时)**:回答里的新判断/新张力/值得连的概念,留痕到相关概念页或笔记
3. **回应**:新一期 🎴 上方用一两句话回应用户的回答

## 记忆文件格式

- **文件名 = 一句话结论**,不打开文件就能看懂
- **正文** = 简短的"为什么" + 证据/来源 + 何时会失效
- 用 Obsidian 双链 `[[目录/文件名]]` 关联相关记忆
- frontmatter:

```yaml
---
type: decision | pitfall | insight | question
created: YYYY-MM-DD
last-verified: YYYY-MM-DD
confidence: high | medium | low   # insight 必填,其余可省
supersedes: 被本条取代的记忆文件名  # 可选
---
```

## 结论速览(SUMMARY.md)

`decisions/`、`pitfalls/`、`insights/` 天生是时间线。`SUMMARY.md` 按主题聚合"现在信什么"而不是"怎么信到这的",末尾双链回所有相关时间线文件。

- **什么时候写**:同一主题下积累 **2 条以上**相关记忆才起条目
- **谁来维护**:写新记忆时顺手检查是否命中已有主题;GC 时也扫一遍
- **和 index.md 的区别**:index.md 全量列表、机械生成;SUMMARY.md 精选聚合、手写综合

## 三条铁律(反信息茧房)

1. **现实优先**:记忆与代码或实测结果冲突时,以现实为准,当场修正并更新 `last-verified`
2. **记忆是先验,不是真理**:依赖 `insights/` 时带着它的 confidence;low confidence 先验证再依赖
3. **人类有否决权**:用户说某条记忆不对,直接删除或修正,不辩护。**不需要定期主动追问"还作数吗"**——AI 一旦按旧记忆做了用户不认可的事,用户会当场说

## 在新仓库安装记忆指针(自安装)

用户在任何仓库说「接入记忆库」「装上记忆」时:把本仓库 `templates/repo-pointer.md` 的内容原样写成该仓库根目录的 `CLAUDE.md`(已存在则追加,不覆盖),提交推送;不能直接推默认分支就开 PR。每周例行任务也会自动巡检用户名下所有仓库并补装缺失的指针。

## Memory GC(垃圾回收,约每周一次)

1. 合并内容重复的记忆:新文件用 `supersedes` 标记,删除旧文件
2. `last-verified` 超过 60 天的 insight,confidence 降一级;已经是 low 的,移入 `questions/` 或删除
3. 没有任何双链指向、也想不出使用场景的记忆,评估后删除
4. `notes/` 特殊:只去重,不自动删。"未被引用"以两个**可实际查证**的信号为准:没有任何双链入链,且从未被工作台抽中(查 `git log -p -- 工作台.md`)。两个信号同时空缺超过 180 天的 note,列进摘要请用户裁决
4a. **手写/粘贴笔记的整理**(必须执行,非可选):用 `git log` 比对自上次 GC 以来**整个仓库**新增/修改的 `.md` 文件,不要只看 `notes/` 目录——重点找游离在 `profile/concepts/decisions/pitfalls/insights/questions/notes` 七个分类目录之外的文件(典型信号:没有 frontmatter、文件名不像结论句)。补齐分类、frontmatter、双链;一个文件混了几个主题就拆分归位,分类或命名不对就直接改名移动,不是只加标注(依据:[[decisions/整理手写笔记允许改写原文而不仅是追加复核记录]])
5. `notes/` 的 `checkable` 内容按上方「notes 的复核」流程处理;`checkable: false` 的一律跳过
6. 扫一遍本周新增的 decisions/pitfalls/insights,凑够 2 条以上但 `SUMMARY.md` 还没条目的,新起或更新
6b. 概念层维护:被 ≥2 篇笔记链到的幽灵概念升格为实体页;挑 1-2 个本周命中的概念补延伸;近义幽灵概念按 aliases 规则合并
7. 重新生成索引并提交;向用户输出一句话摘要(动了什么、为什么)
8. 无事可做则保持沉默,**不要为了排期而主动问"这条还作数吗"**(见「三条铁律」第 3 条)
