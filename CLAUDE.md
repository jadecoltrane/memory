# CLAUDE.md — AI 协作与记忆库规则

本仓库(`jadecoltane/memory`)**就是记忆库本身**:纯 Markdown + Obsidian 双链,不绑定任何工具。
读写它的 AI 都遵循这同一套规则(当前只接入 Claude;库本身是纯 Markdown,任何工具将来都能接入)。文件名就是一句话结论,正文是简短的"为什么"。

**索引已自动化**:`.github/workflows/rebuild-index.yml` 会在任何推送到 main 且改动了记忆文件时自动重建 `index.md`,不需要人或 AI 记得手动跑脚本——包括用户直接在 Obsidian 里写完推送、完全没有 AI 参与的情况。AI 手动跑 `scripts/build_memory_index.py` 仍然保留(能在推送前就看到一致的状态),但不再是索引不过期的唯一保障。

## 开工时(必做)

1. 读根目录 `index.md`(索引,每条记忆一行)
2. 只按需打开与当前任务相关的记忆文件,不要全量阅读

## 收工时(必做)

判断本次会话是否产生了值得沉淀的记忆。**该往哪个目录写、写不写,查根目录 `RESOLVER.md`**(判断逻辑单独维护在那,这里不重复)。

写完后运行 `python3 scripts/build_memory_index.py` 重新生成索引;若新记忆和已有记忆同属一个主题(2 条以上),同步维护 `SUMMARY.md`(见「结论速览」一节);记忆、索引、速览一起提交。

**写入必须告知**:任何时候写入或修改了记忆(自动沉淀或用户触发都一样),必须在给用户的回复末尾用一两行明确说出来,格式如:「🧠 已写入记忆:`decisions/xxx`」。禁止悄悄写入。

## 我的收藏(notes/)——用户点名的一切都进这里

`notes/` 存用户认证过"值得留"的知识点,与上面四类的区别:**永远不要自动写入,只响应用户的明确触发**。

- **触发词**:用户说「记下来」「存一下」「收藏这条」「写进记忆库」「remember this」或同义表达时:
  1. 把刚才讨论的内容提炼成一条笔记(不是全文粘贴,是提炼)
  2. **一律写进 `notes/`,不做分流**——用户要求自己点名的内容全部集中在 notes/ 一处可见,无论是知识、决定还是感想。若该内容同时构成对 AI 未来行为的约束(如「以后都用 X」),AI 另在 `decisions/` 补一条自己的工作记忆并与该 note 双链,两边各司其职。notes/ 下允许自由创建子文件夹分类,索引脚本会递归收录
  3. 写入本仓库(`jadecoltane/memory`)、重跑索引、提交并推送到 main。**"记下来"永远指写进这个 git 仓库**,而不是会话内的短期记忆或任何 AI 产品自带的记忆功能——只有仓库里的记忆是持久、可迁移、Obsidian 可见的
  4. 如果当前会话没有本仓库,先获取它(Claude Code 远程会话用 add_repo 添加 jadecoltane/memory;本地会话直接操作用户磁盘上的 clone),实在无法访问时把写好的记忆全文输出给用户,请用户手动保存
- **格式**:frontmatter 用 `type: note`,加一行 `source:`(一句话说明来自哪次讨论);正文照常是提炼内容 + 双链
- **先判断是否"可查证"**:写入前先分类——
  - **事实性/可查证**(数据、健康、历史、科学结论、他人转述的客观陈述等):走下面的 verified + 复核流程
  - **主观性/不可查证**(个人想法、感受、散文、创作、经历叙述、审美偏好):frontmatter 加 `checkable: false`,**不设 verified 字段,永远不进复核队列**——这类内容没有"对不对",核实它是荒谬的,它的价值就是"这是你此刻的想法",原样留存即可
- **verified 标记**(仅 `checkable` 内容需要):内容来自用户自己亲历/验证过的判断 → `verified: true`;转述自其他 AI、网络、他人、未经交叉核实 → `verified: false`。拿不准时默认 `false`,宁可多复核不要漏
- **GC 规则**:notes 只做去重合并,**不自动降级、不自动删除**——它们是用户亲自认证"值得留",删除权只在用户

### notes 的复核(反茧房,防止未经查证的内容被当真理囤积)

**仅对 `checkable` 内容生效**;`checkable: false` 的笔记跳过本节全部流程。

**复核节奏(不是查过一次就永远每月重查,按上次结果动态调整)**:

- 从未复核过的 `verified: false` note:30 天内必查
- 上次结果是「支持」:间隔延长到 180 天再查(证据不会短期内变化,月月重查是浪费)
- 上次结果是「存疑」或「找不到依据」:下次 GC(约一周后)继续查,直到用户裁决或结论转为「支持」

判定到期后:

1. **只用工具实际检索到的结果,禁止凭训练记忆编造引用**——所有引用链接必须来自 WebSearch/WebFetch 当次调用返回的真实 URL,没查到就写"未找到可靠来源",不要硬凑一个
2. 判断结论:支持 / 有争议或部分不准 / 找不到依据
3. **绝不修改或删除原文**——在文件末尾追加一个 `## 复核记录` 区块(没有则新建,有则追加新条目),格式:`- YYYY-MM-DD:结论(支持/存疑/待考),简述依据 + 真实来源链接,并注明下次复核的大致时间`
4. 复核发现「存疑」或「找不到依据」的条目,列入当周摘要明确提醒用户,给出理由,由用户决定是否修改、标记 `verified: true`(采信)或删除
5. 复核后是「支持」的,静默追加记录即可,不用打扰用户
6. `verified: true` 的 note 不做真实性复核(用户已认证),仍适用上面的「180 天未引用」使用度检查

### 让 notes 活起来:概念关联与反思

不只是攒文件,笔记之间要长出联想、张力、甚至矛盾——这是双链图谱存在的意义。

- **每次新写一条 note 时**(不管是响应用户触发写的,还是发现用户自己在 Obsidian 手写、还没建任何链接的笔记):除了当次讨论里提到的关联,额外花一点心思扫一遍 `notes/` 里主题相近的旧笔记(用 index.md 快速定位候选,不用全读),看看有没有值得连的——包括**观点上呼应或冲突**的,不只是主题相同的。找到了就加双链,双向都加(新笔记链旧笔记,也回旧笔记里补一条链回来)
- **用户自己在 Obsidian 里手写的笔记,格式不限、不需要 frontmatter**;只要它被推送到了 GitHub,下面两个时机之一就会处理它,自动补齐分类、frontmatter、双链:
  1. **每次 GC**(约每周一次):用 `git log` 比对自上次 GC 以来有变更的文件,扫一遍这些新增/修改的 notes(包括用户直接手写、没有经过对话触发的),该建的双链建上,同时顺手做一遍"张力扫描"——看这些笔记之间、以及和旧笔记之间,有没有互相矛盾或前后判断不一致的地方(比如一条说"喜欢 X 的风格",另一条流露出相反的偏好)。发现了不要替用户调和,**原样列进当周摘要,用一句话提出问题**,不给结论,思考的过程留给用户自己走
  2. **用户随时说**「帮我梳理一下 notes」「看看我最近的笔记有什么关联」,不用等到下周,当场触发处理
- 换句话说:**你在 Obsidian 里想怎么写就怎么写,唯一要做的是让 Obsidian Git 插件把它推送到 GitHub**——推送是我能看到它的前提,没推送等于我不知道这条笔记存在
- 找不到有价值的关联或张力时不要为了完成任务而硬凑,宁可什么都不说

### 图片(截图等)

用户发图片(如小红书截图)并说「记下来」时:

1. 原图保存到 `notes/assets/YYYY-MM-DD-简短描述.png`(或原格式,不要转码压缩)
2. 对应的 note 里用 `![说明](assets/文件名)` 嵌入原图,**并且必须额外写一段文字提炼**(截图讲了什么、要点是什么)——图片本身无法被索引和 grep,未来 AI 靠这段文字才知道这条记忆在讲什么,图片是佐证材料而非索引依据
3. 单张截图通常几百 KB 到几 MB,GitHub 单文件上限 100MB,正常使用无需顾虑;但不要把长视频或高清大图仓库当网盘用

## 记忆文件格式

- **文件名 = 一句话结论**,必须不打开文件就能看懂(索引里只展示文件名)
- **正文** = 简短的"为什么" + 证据/来源 + 何时会失效
- 用 Obsidian 双链 `[[目录/另一条记忆的文件名]]` 关联相关记忆
- frontmatter:

```yaml
---
type: decision | pitfall | insight | question
created: YYYY-MM-DD
last-verified: YYYY-MM-DD
confidence: high | medium | low   # insight 必填,其余可省
supersedes: 被本条取代的记忆文件名  # 可选
---
```

## 结论速览(SUMMARY.md)

`decisions/`、`pitfalls/`、`insights/` 天生是时间线:一条条独立文件,靠 `supersedes` 演进。时间线的问题是"想知道现在对某件事的判断是什么"得把好几篇翻出来自己拼。`SUMMARY.md` 是给这个问题的解:按主题聚合"现在信什么",不是按时间讲"怎么信到这的"——后者永远去时间线文件里查,`SUMMARY.md` 只给结论,末尾双链回所有相关时间线文件保留可追溯性。

- **什么时候写**:同一主题下积累了 **2 条以上**相关的 decisions/pitfalls/insights 才起条目,数量不够就不写——不为还没出现的复杂度预先建索引
- **谁来维护**:写新记忆时顺手检查是否命中已有主题,命中就更新对应段落;GC 时也扫一遍有没有该合并的新主题
- **和 index.md 的区别**:index.md 是全量列表(每条记忆一行,机械生成,勿手改);SUMMARY.md 是精选聚合(手写综合,少而准,允许并且需要手改)

## 三条铁律(反信息茧房)

1. **现实优先**:记忆与代码或实测结果冲突时,以现实为准,并当场修正该记忆、更新 `last-verified`。记忆是缓存,缓存必须能失效。
2. **记忆是先验,不是真理**:依赖 `insights/` 里的判断时带着它的 confidence;low confidence 的记忆在依赖前先验证。
3. **人类有否决权**:用户说某条记忆不对,直接删除或修正,不要辩护。这条同时覆盖"记忆本身错了"和"记忆过时了、用户偏好变了"两种情况——**不需要定期主动追问用户"还作数吗"**,AI 一旦按旧记忆做了用户不认可的事,用户会当场说,那时再改比排期表更及时,也不平白打扰用户。

## 在新仓库安装记忆指针(自安装)

用户在任何仓库的会话里说「接入记忆库」「装上记忆」或同义表达时:
把本仓库 `templates/repo-pointer.md` 的内容原样写成该仓库根目录的 `CLAUDE.md`(若已存在 CLAUDE.md,则把指针内容追加进去而不是覆盖),提交并推送;不能直接推默认分支就开 PR。装完当场生效,不需要用户手动复制任何东西。每周例行任务也会自动巡检用户名下所有仓库并补装缺失的指针。

## Memory GC(垃圾回收,约每周一次)

1. 合并内容重复的记忆:新文件用 `supersedes` 标记,删除旧文件
2. `last-verified` 超过 60 天的 insight,confidence 降一级;已经是 low 的,移入 `questions/` 或删除
3. 没有任何双链指向、也想不出使用场景的记忆,评估后删除
4. `notes/` 特殊:只去重,不自动删;但超过 180 天未被任何记忆或任务引用的 note,列进摘要请用户裁决
5. `notes/` 的 `checkable` 内容按上方「notes 的复核」流程处理;`checkable: false` 的一律跳过
6. 扫一遍本周新增的 decisions/pitfalls/insights,看有没有哪个主题下相关记忆凑够了 2 条以上、但 `SUMMARY.md` 里还没有对应条目——有就新起或更新一条(见「结论速览」一节)
7. 重新生成索引并提交;向用户输出一句话摘要(动了什么、为什么)
8. 无事可做则保持沉默,不打扰用户;**不要为了排期而主动问用户"这条还作数吗"**——过时的决定/判断由用户在实际用到时当场纠正(见「三条铁律」第 3 条),GC 不做这类打扰式追问
