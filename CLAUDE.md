# CLAUDE.md — AI 协作与记忆库规则

`jadecoltrane/memory` 就是记忆库本身:数据层纯 Markdown + Git(可迁移,任何 AI 都能接入),使用层 Obsidian 插件随便用(Dataview、Git 等)。当前只接入 Claude。

**兜底优先**:可靠性相关的东西(失败告警、机器校验、边界情况、备份)必须**预先**设计,不等出事再补;懒加载只允许用在内容组织,不适用于安全网。

**方案设计优先省 token**:几种做法效果相当时选更省 token(读/维护成本更低)的,但不能拿效果去换——本文件和各细则文件、自动化 prompt 都按这个标准写,发现冗余可以精简,但不能牺牲可执行性。

**细则按需加载**(本文件只留触发条件和指针,命中才读对应文件,平时聊天不加载):

| 文件 | 什么时候读 |
|---|---|
| `meta/RESOLVER.md` | 收工判断写不写记忆、写哪;含记忆文件格式和 SUMMARY 维护规则 |
| `meta/NOTES.md` | 触发词命中要写一条 note 时;整理 notes 时 |
| `meta/CONCEPTS.md` | 动 `concepts/` 时(建页、补延伸、维护开放问题) |
| `meta/GC.md` | 执行每周 GC 或处理 note 复核到期时 |
| `meta/WORKBENCH.md` | 处理工作台相关任务时 |

**四个自动化,不需要 AI 手动补**:
- `rebuild-index.yml`:推送到 main 且改了记忆文件就自动重建 `meta/index.md`(AI 仍可手动跑 `meta/scripts/build_memory_index.py` 提前核对)
- `weekly-gc.yml`:每周一 09:00(北京时间)云端跑 `meta/GC.md` 的步骤;自上次 GC 零实质性新提交就跳过整次调用
- `daily-workbench.yml`:每天 05:00(北京时间)云端生成新一期 `工作台.md`
- `lint.yml`:每次推送跑 `meta/scripts/lint_memory.py` 查断链和 frontmatter 完整性,失败邮件通知(幽灵概念是设计允许的,不算断链)。AI 写完记忆建议本地先跑一遍

## 开工时(必做)

1. 读 `meta/index.md`(索引,每条记忆一行),顺手扫一眼 profile/ 条目
2. 只按需打开与当前任务相关的记忆文件,不要全量阅读

## 回答问题时

先看 `meta/index.md` 和相关的 `notes/`、`concepts/` 里有没有直接相关的内容——有就基于笔记内容摘要作答,并注明引用了哪篇笔记(纯文本路径);没有就按常规知识正常回答,不用硬凑。这条只影响"怎么答问题",写不写入记忆仍按 `meta/RESOLVER.md`。

## 收工时(必做)

判断本次会话是否产生了值得沉淀的记忆。**写不写、往哪写、什么格式,全按 `meta/RESOLVER.md`**(含写完后的索引重建和 SUMMARY 检查)。

**写入必须告知**:任何时候写入或修改了记忆(自动沉淀或用户触发都一样),必须在给用户的回复末尾明确说出来,格式如:「🧠 已写入记忆:`decisions/xxx`」。禁止悄悄写入。

## 用户画像(profile/)——记录"用户是谁",主动维护,无需触发词

记忆库优先记录用户,而不是记录它自己。`profile/` 按主题一页,按需新增(职业、教育、人际、健康、价值观……),写入照样按「写入必须告知」执行。

- **客观事实**(职业、城市、经历、家庭等)→ 说一次就记,不用等第二次
- **AI 归纳的软特质**(偏好、风格、性格倾向)→ 用户明确表达过,或不同场合重复出现 ≥2 次才写,防止过度解读;每条附一句依据(哪次对话、哪篇笔记)和日期;特质变了就地更新并注明,不另开新文件
- **不做双链、不进图谱**:引用其他记忆用纯文本路径。图谱只属于用户的知识(notes + concepts),本地 Obsidian 图谱过滤器同时排除 profile/decisions/pitfalls/insights/questions 和 `meta/` 基础设施文件
- frontmatter:`type: profile`、`updated: YYYY-MM-DD`
- 用户否决权适用:说哪条不对,当场改,不辩护

## 我的收藏(notes/)——只响应用户,永远不自动写入

`notes/` 只放**文章、知识点、用户自己的想法/创作/经历**——不放 AI 协作规则、对 AI 行为的约束(那些归 `decisions/` 或 `profile/`)。

- **触发词**:「记下来」「存一下」「收藏这条」「写进记忆库」「remember this」→ 先按 `meta/RESOLVER.md` 判断内容性质,确属 notes 的,**读 `meta/NOTES.md` 按里面的流程执行**(提炼、checkable/verified 标记、图片处理、概念双链)
- **"记下来"永远指写进这个 git 仓库并推送 main**,不是会话内的短期记忆或任何 AI 产品自带的记忆功能。当前会话没有本仓库时先获取(远程会话用 add_repo 添加 jadecoltrane/memory;本地会话直接操作用户磁盘上的 clone),实在无法访问时把写好的内容整段输出给用户手动保存
- **推送是前提**:用户在 Obsidian 手写的笔记,没推送到 GitHub 等于 AI 不知道它存在。散装手写笔记的整理在每周 GC 或用户说「帮我梳理一下 notes」时做(流程见 `meta/NOTES.md` 和 `meta/GC.md`)

## 概念层(concepts/)

写 note 时正文概念出现处挂 `[[concepts/xxx]]`(页面不存在也照写,幽灵节点零成本)——这是录入时唯一要做的动作;其余一切(建页时机、页面结构、aliases、开放问题生命周期)**动 `concepts/` 前读 `meta/CONCEPTS.md`**。

## 三条铁律(反信息茧房)

1. **现实优先**:记忆与代码或实测结果冲突时,以现实为准,当场修正并更新 `last-verified`
2. **记忆是先验,不是真理**:依赖 `insights/` 时带着它的 confidence;low confidence 先验证再依赖
3. **人类有否决权**:用户说某条记忆不对,直接删除或修正,不辩护。**不需要定期主动追问"还作数吗"**——AI 一旦按旧记忆做了用户不认可的事,用户会当场说

## 在新仓库安装记忆指针(自安装)

用户在任何仓库说「接入记忆库」「装上记忆」时:把本仓库 `meta/repo-pointer.md` 的内容原样写成该仓库根目录的 `CLAUDE.md`(已存在则追加,不覆盖),提交推送;不能直接推默认分支就开 PR。
