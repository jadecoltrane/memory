# CLAUDE.md — AI 协作与记忆库规则

本仓库(`jadecoltane/memory`)**就是记忆库本身**:数据层是纯 Markdown + Git,保证可迁移、任何 AI 将来都能接入;使用层上 **Obsidian 插件放开用**(Dataview、Obsidian Git 等)——不绑定的是数据,不是使用体验。AI 生成的页面可以依赖插件渲染动态内容。
读写它的 AI 都遵循这同一套规则(当前只接入 Claude)。文件名就是一句话结论,正文是简短的"为什么"。

**隐私边界**:本仓库包含健康、心理、生活偏好等个人隐私,必须保持 GitHub **private**。任何 AI 不得把仓库转公开、不得把记忆内容发布到任何外部服务;引用记忆内容回答问题仅限用户本人的会话。(建库时曾误设为公开,2026-07-04 发现并转私有,见 [[pitfalls/记忆库仓库曾默认公开导致隐私内容对外暴露]])

**设计原则:兜底优先(用户 2026-07-04 明确要求)**:可靠性相关的东西——失败告警、机器校验、边界情况、备份——必须**预先**设计,不等出了事再补;"复杂度还没出现"不是不设计的理由。允许懒加载的只有内容组织(概念页第二次命中才建、SUMMARY 凑够 2 条才起条目),因为那是内容,不是安全网。此前 AI 自定的"不为未出现的复杂度预建"取向已被用户否决,仅在内容组织范围内保留。

**索引已自动化**:`.github/workflows/rebuild-index.yml` 会在任何推送到 main 且改动了记忆文件时自动重建 `index.md`,不需要人或 AI 记得手动跑脚本——包括用户直接在 Obsidian 里写完推送、完全没有 AI 参与的情况。AI 手动跑 `scripts/build_memory_index.py` 仍然保留(能在推送前就看到一致的状态),但不再是索引不过期的唯一保障。

**每周 GC 已自动化**:`.github/workflows/weekly-gc.yml` 每周一 09:00(北京时间)在云端执行「Memory GC」一节的流程和指针巡检,并直接提交推送——AI 不需要另行安排每周 GC;用户临时要求时按需手动执行即可。指针巡检部分依赖仓库 secret `REPO_PATROL_PAT`(跨仓库写权限的 PAT),未配置时自动跳过。

**规则执行有机器兜底**:`.github/workflows/lint.yml` 在每次推送时跑 `scripts/lint_memory.py`,检查断链和 frontmatter 完整性(幽灵概念是设计允许的,不算断链),失败会亮红灯并邮件通知用户——规则写给 AI,但监督不能只靠 AI 自觉(真实案例:supersede 删旧文件后,5 处反向双链全都忘了更新,是 lint 抓出来的)。AI 写完记忆后建议本地跑一遍 lint 再提交。

## 开工时(必做)

1. 读根目录 `index.md`(索引,每条记忆一行)
2. 只按需打开与当前任务相关的记忆文件,不要全量阅读

## 收工时(必做)

判断本次会话是否产生了值得沉淀的记忆。**该往哪个目录写、写不写,查根目录 `RESOLVER.md`**(判断逻辑单独维护在那,这里不重复)。

写完后运行 `python3 scripts/build_memory_index.py` 重新生成索引;若新记忆和已有记忆同属一个主题(2 条以上),同步维护 `SUMMARY.md`(见「结论速览」一节);记忆、索引、速览一起提交。

**写入必须告知**:任何时候写入或修改了记忆(自动沉淀或用户触发都一样),必须在给用户的回复末尾用一两行明确说出来,格式如:「🧠 已写入记忆:`decisions/xxx`」。禁止悄悄写入。

## 用户画像(profile/)——AI 主动记录"用户是谁"

**记忆库优先记录用户,而不是记录它自己**(用户 2026-07-04 明确要求)。`profile/` 按主题一页,**主题页按需新增,不限于已有几页**——职业与工作、教育背景、人际关系、健康状况、价值观……任何关于用户本人的稳定信息都可以起新页。**不需要触发词,AI 主动维护**,写入照样按「写入必须告知」执行。

- **什么时候写,分两档**:
  - 用户陈述了关于自己的**客观事实**(职业、城市、经历、家庭等)→ 说一次就记,不用等第二次
  - AI 自己**归纳的软特质**(偏好、风格、性格倾向)→ 用户明确表达过,或在不同场合重复出现 ≥2 次才写,防止过度解读
- **怎么写**:每条特质一个要点,附一句依据(哪次对话、哪篇笔记观察到)和日期;特质变了就地更新并注明,不另开新文件
- **不做双链、不进图谱**(用户 2026-07-04 明确否决了"画像也要进图谱"):profile 正文不挂 `[[concepts/xxx]]`、不与其他文件双链,引用其他记忆一律用纯文本路径。**图谱只属于用户的知识(notes + concepts),AI 自动记忆不掺和**;本地 Obsidian 的图谱过滤器同时把 profile/decisions/pitfalls/insights/questions 和根目录基础设施文件排除在外,双保险
- **和其他目录的分工**:notes 等用户点名才写;insights 是对世界与工作的判断;profile 只关于用户本人。构成对 AI 行为硬约束的准则,照旧在 decisions/ 补一条工作记忆
- frontmatter:`type: profile`、`updated: YYYY-MM-DD`
- **开工时**连同 index.md 一起顺手扫一眼 profile/——它是"用户是谁"的最短路径
- 用户否决权当然适用:说哪条不对,当场改,不辩护

## 我的收藏(notes/)——用户点名的一切都进这里

`notes/` 存用户认证过"值得留"的知识点,与上面四类的区别:**永远不要自动写入,只响应用户的明确触发**。

- **触发词**:用户说「记下来」「存一下」「收藏这条」「写进记忆库」「remember this」或同义表达时:
  1. 把刚才讨论的内容提炼成一条笔记(不是全文粘贴,是提炼)
  2. **一律写进 `notes/`,不做分流**——用户要求自己点名的内容全部集中在 notes/ 一处可见,无论是知识、决定还是感想。若该内容同时构成对 AI 未来行为的约束(如「以后都用 X」),AI 另在 `decisions/` 补一条自己的工作记忆并与该 note 双链,两边各司其职。notes/ 下允许自由创建子文件夹分类,索引脚本会递归收录
  3. 写入本仓库(`jadecoltane/memory`)、重跑索引、提交并推送到 main。**"记下来"永远指写进这个 git 仓库**,而不是会话内的短期记忆或任何 AI 产品自带的记忆功能——只有仓库里的记忆是持久、可迁移、Obsidian 可见的
  4. 如果当前会话没有本仓库,先获取它(Claude Code 远程会话用 add_repo 添加 jadecoltane/memory;本地会话直接操作用户磁盘上的 clone),实在无法访问时把写好的记忆全文输出给用户,请用户手动保存
- **格式**:frontmatter 用 `type: note`,加一行 `source:`(一句话说明来自哪次讨论);正文照常是提炼内容 + 双链
- **先判断是否"可查证"**:写入前先分类——
  - **事实性/可查证**(数据、健康、历史、科学结论、他人转述的客观陈述等):走下面的 verified + 复核流程
  - **主观性/不可查证**(个人想法、感受、散文、创作、经历叙述、审美偏好):frontmatter 加 `checkable: false`,**不设 verified 字段,永远不进复核队列**——这类内容没有"对不对",核实它是荒谬的,它的价值就是"这是你此刻的想法",原样留存即可
- **verified 标记**(仅 `checkable` 内容需要):内容来自用户自己亲历/验证过的判断 → `verified: true`;转述自其他 AI、网络、他人、未经交叉核实 → `verified: false`。拿不准时默认 `false`,宁可多复核不要漏
- **GC 规则**:notes 只做去重合并,**不自动降级、不自动删除**——它们是用户亲自认证"值得留",删除权只在用户

### notes 的复核(反茧房,防止未经查证的内容被当真理囤积)

**仅对 `checkable` 内容生效**;`checkable: false` 的笔记跳过本节全部流程。

**复核节奏(不是查过一次就永远每月重查,按上次结果动态调整)**:

- 从未复核过的 `verified: false` note:30 天内必查
- 上次结果是「支持」:间隔延长到 180 天再查(证据不会短期内变化,月月重查是浪费)
- 上次结果是「存疑」或「找不到依据」:下次 GC(约一周后)继续查,直到用户裁决或结论转为「支持」

判定到期后:

1. **只用工具实际检索到的结果,禁止凭训练记忆编造引用**——所有引用链接必须来自 WebSearch/WebFetch 当次调用返回的真实 URL,没查到就写"未找到可靠来源",不要硬凑一个
2. 判断结论:支持 / 有争议或部分不准 / 找不到依据
3. **绝不修改或删除原文**——在文件末尾追加一个 `## 复核记录` 区块(没有则新建,有则追加新条目),格式:`- YYYY-MM-DD:结论(支持/存疑/待考),简述依据 + 真实来源链接,并注明下次复核的大致时间`
4. 复核发现「存疑」或「找不到依据」的条目,列入当周摘要明确提醒用户,给出理由,由用户决定是否修改、标记 `verified: true`(采信)或删除
5. 复核后是「支持」的,静默追加记录即可,不用打扰用户
6. `verified: true` 的 note 不做真实性复核(用户已认证),仍适用上面的「180 天未引用」使用度检查

### 让 notes 活起来:概念关联与反思

不只是攒文件,笔记之间要长出联想、张力、甚至矛盾——这是双链图谱存在的意义。

- **每次新写一条 note 时**(不管是响应用户触发写的,还是发现用户自己在 Obsidian 手写、还没建任何链接的笔记):除了当次讨论里提到的关联,额外花一点心思扫一遍 `notes/` 里主题相近的旧笔记(用 index.md 快速定位候选,不用全读),看看有没有值得连的——包括**观点上呼应或冲突**的,不只是主题相同的。找到了就加双链,双向都加(新笔记链旧笔记,也回旧笔记里补一条链回来)
- **用户自己在 Obsidian 里手写的笔记,格式不限、不需要 frontmatter,甚至可能随手扔在仓库根目录、没进任何 `notes/` 子文件夹、内容混了好几个不相关主题**;只要它被推送到了 GitHub,下面两个时机之一就会处理它,自动补齐分类、frontmatter、双链:
  1. **每次 GC**(约每周一次):用 `git log` 比对自上次 GC 以来**整个仓库**新增/修改的 `.md` 文件(不要只看 `notes/` 目录内部——重点排查有没有游离在 `profile/concepts/decisions/pitfalls/insights/questions/notes` 这七个已知分类目录之外的文件,那就是还没归类的手写内容,典型信号是没有 frontmatter、文件名不像结论句)。对这些文件和 `notes/` 内新增/修改的笔记一视同仁:该建的双链建上,同时顺手做一遍"张力扫描"——看这些笔记之间、以及和旧笔记之间,有没有互相矛盾或前后判断不一致的地方(比如一条说"喜欢 X 的风格",另一条流露出相反的偏好)。发现了不要替用户调和,**原样列进当周摘要,用一句话提出问题**,不给结论,思考的过程留给用户自己走。**这一步不是只加标注**——按 [[decisions/整理手写笔记允许改写原文而不仅是追加复核记录]] 的授权,一个文件混了几个主题就拆成几篇、分类或文件名不对就重新归位改名,直接动手改原文,不用等用户开口
  2. **用户随时说**「帮我梳理一下 notes」「看看我最近的笔记有什么关联」,不用等到下周,当场触发处理
- 换句话说:**你在 Obsidian 里想怎么写就怎么写,唯一要做的是让 Obsidian Git 插件把它推送到 GitHub**——推送是我能看到它的前提,没推送等于我不知道这条笔记存在
- 找不到有价值的关联或张力时不要为了完成任务而硬凑,宁可什么都不说

### 图片(截图等)

用户发图片(如小红书截图)并说「记下来」时:

1. 原图保存到 `notes/assets/YYYY-MM-DD-简短描述.png`(或原格式,不要转码压缩)
2. 对应的 note 里用 `![说明](assets/文件名)` 嵌入原图,**并且必须额外写一段文字提炼**(截图讲了什么、要点是什么)——图片本身无法被索引和 grep,未来 AI 靠这段文字才知道这条记忆在讲什么,图片是佐证材料而非索引依据
3. 单张截图通常几百 KB 到几 MB,GitHub 单文件上限 100MB,正常使用无需顾虑;但不要把长视频或高清大图仓库当网盘用

## 概念层(concepts/)——图谱的枢纽节点

笔记是文档,概念才是思考的单位。`concepts/` 每页一个原子概念(如「水缸」「客服模式」),让图谱长出概念连接而不是文档互相引用。页面结构四段:一句话定义(用用户自己的话)、出现在(链回笔记)、相关概念(概念互链)、延伸+开放问题(AI 补充的外部知识和联想钩子)。

**双链规则**:笔记正文里的双链优先指向概念页(`[[concepts/某概念]]`),**笔记之间不直接互链**——两篇笔记通过共同概念在图谱里相连,不要为了关联而把一篇笔记硬塞进另一篇的正文。例外:两篇确实是同一论证的上下篇时可以直链。

**同义词用 aliases 收编,不开新节点**:同一个意思换个说法就是一个新幽灵节点,概念多了图谱会碎成一地近义节点。对策:概念页 frontmatter 加 `aliases: [同义表述1, 同义表述2]`;写双链前先扫一眼 index 的概念列表和已有 aliases,能挂靠已有概念就不要造新名字。GC 时发现近义的幽灵概念(幽灵与幽灵、幽灵与实体页),合并:保留一个正名,其余收进实体页的 aliases,并改写旧链接指向正名。

**懒加载(控制 token 成本,重活挪到"用户在场且想要"的时刻)**:

1. **录入时只做最小动作**:扫一眼 index 的概念列表,正文概念出现处写 `[[concepts/xxx]]`——概念页不存在也照写。图谱配置 `hideUnresolved: false`,未创建的链接以幽灵节点显示,概念在图谱里出现是零成本的
2. **第二次命中才建页**:一个概念被第二篇笔记链到时才创建实体页,且只写「定义+出现在」两段,延伸区留空(与 SUMMARY 的"凑够 2 条才起条目"同一原则)
3. **延伸和开放问题只在两个时机写**:用户主动要求展开某个概念时(现场查现场写);每周 GC 挑 1-2 个本周被新笔记命中的概念批量补。**绝不在录入时顺手写延伸**

概念页 frontmatter 用 `type: concept`;延伸区每条标注 [事实]/[推论]/[观点],凭训练知识写的必须注明"未联网核查",禁止编造引用链接(同 notes 复核规则)。
## 工作台(每日生成)

根目录 `工作台.md` 是记忆库的出口:让旧记忆主动找上门,核心是提取练习而不是重读。每天由定时任务重新生成三个 AI 板块,其余部分(顶部 dataviewjs 横幅与统计卡、📥 最近输入、👻 幽灵概念)是 Dataview 实时渲染,**生成时原样保留,永远不要改动**。格式必须保持现有的 callout 卡片语法(`[!draw]` `[!seed]` `[!ask]`);frontmatter 保留 `cssclasses: workbench`,但每次生成**必须把 `generated:` 更新为当天日期(北京时间)**——横幅的心跳报警靠它判断每日任务是否还活着(超过 48 小时未更新显红),这个字段漏更会误报,它同时也是"系统静默挂掉"唯一的可见信号。

每日生成三个板块的规则:

- **🎴 今日抽卡**:从 `notes/` 挑"最久没被工作台抽过"的(避免连续重复,可查 git log 里工作台的历史版本),出 1-3 道自测题——考核心结构而不是细节,题目不剧透答案。**抽几篇随体量扩容**:默认每天 1 篇,notes 超过 100 篇后每天 2 篇、超过 250 篇后每天 3 篇——抽卡是旧笔记唯一的复活通道,体量涨了通道不加宽,大部分记忆就永远躺在死区
- **🌱 今日延伸**:挑一个概念页(优先最近被新笔记链到的、或延伸区还空着的),补充一条外部知识,标注 [事实]/[推论]/[观点],凭训练知识写的注明未联网核查;同时把这条延伸同步写进该概念页的延伸区(工作台是易逝的,概念页是积累的)
- **❓ 今天的问题**:优先从概念页的开放问题、或笔记之间的张力(矛盾、不一致)里出;问具体的、指向用户自身经验的问题,不问抽象大词

**生成前必做**:读上一版工作台 ❓ 区块里用户写的回答(✍️ 虚线框内)。有回答时,依次做三件事:

1. **归档(无条件)**:把问题原文+用户回答原文,追加到 `notes/问答/YYYY-MM 工作台问答.md`(按月一个文件,frontmatter `type: note`、`checkable: false`;首次创建时补 frontmatter)。用户亲手写的回答是 notes 级内容,适用 notes 的规则:永不自动删除。工作台本身每天原地覆盖,归档是回答唯一的持久落点,**先归档再覆盖,顺序不能反**
2. **沉淀(有价值时)**:回答里出现新判断、新张力或值得连的概念时,在相关概念页或笔记里留痕、加双链
3. **回应**:在新一期 🎴 上方用一两句话回应用户的回答——被读到、被回应,用户才会持续写

## 记忆文件格式

- **文件名 = 一句话结论**,必须不打开文件就能看懂(索引里只展示文件名)
- **正文** = 简短的"为什么" + 证据/来源 + 何时会失效
- 用 Obsidian 双链 `[[目录/另一条记忆的文件名]]` 关联相关记忆
- frontmatter:

```yaml
---
type: decision | pitfall | insight | question
created: YYYY-MM-DD
last-verified: YYYY-MM-DD
confidence: high | medium | low   # insight 必填,其余可省
supersedes: 被本条取代的记忆文件名  # 可选
```

## 结论速览(SUMMARY.md)

`decisions/`、`pitfalls/`、`insights/` 天生是时间线:一条条独立文件,靠 `supersedes` 演进。时间线的问题是"想知道现在对某件事的判断是什么"得把好几篇翻出来自己拼。`SUMMARY.md` 是给这个问题的解:按主题聚合"现在信什么",不是按时间讲"怎么信到这的"——后者永远去时间线文件里查,`SUMMARY.md` 只给结论,末尾双链回所有相关时间线文件保留可追溯性。

- **什么时候写**:同一主题下积累了 **2 条以上**相关的 decisions/pitfalls/insights 才起条目,数量不够就不写(内容条目的懒加载,见开头「设计原则」——兜底和安全网不适用这条,必须预建)
- **谁来维护**:写新记忆时顺手检查是否命中已有主题,命中就更新对应段落;GC 时也扫一遍有没有该合并的新主题
- **和 index.md 的区别**:index.md 是全量列表(每条记忆一行,机械生成,勿手改);SUMMARY.md 是精选聚合(手写综合,少而准,允许并且需要手改)

## 三条铁律(反信息茧房)

1. **现实优先**:记忆与代码或实测结果冲突时,以现实为准,并当场修正该记忆、更新 `last-verified`。记忆是缓存,缓存必须能失效。
2. **记忆是先验,不是真理**:依赖 `insights/` 里的判断时带着它的 confidence;low confidence 的记忆在依赖前先验证。
3. **人类有否决权**:用户说某条记忆不对,直接删除或修正,不要辩护。这条同时覆盖"记忆本身错了"和"记忆过时了、用户偏好变了"两种情况——**不需要定期主动追问用户"还作数吗"**,AI 一旦按旧记忆做了用户不认可的事,用户会当场说,那时再改比排期表更及时,也不平白打扰用户。

## 在新仓库安装记忆指针(自安装)

用户在任何仓库的会话里说「接入记忆库」「装上记忆」或同义表达时:
把本仓库 `templates/repo-pointer.md` 的内容原样写成该仓库根目录的 `CLAUDE.md`(若已存在 CLAUDE.md,则把指针内容追加进去而不是覆盖),提交并推送;不能直接推默认分支就开 PR。装完当场生效,不需要用户手动复制任何东西。每周例行任务也会自动巡检用户名下所有仓库并补装缺失的指针。

## Memory GC(垃圾回收,约每周一次)

1. 合并内容重复的记忆:新文件用 `supersedes` 标记,删除旧文件
2. `last-verified` 超过 60 天的 insight,confidence 降一级;已经是 low 的,移入 `questions/` 或删除
3. 没有任何双链指向、也想不出使用场景的记忆,评估后删除
4. `notes/` 特殊:只去重,不自动删。"未被引用"以两个**可实际查证**的信号为准:没有任何双链入链,且从未被工作台抽中(用 `git log -p -- 工作台.md` 查抽卡历史)——"AI 会话里用没用到"没有记录机制,不作为判断依据。两个信号同时空缺超过 180 天的 note,列进摘要请用户裁决
4a. **手写/粘贴笔记的整理**(完整规则见「我的收藏」→「让 notes 活起来」一节,这里是 GC 必须执行的提醒,不是可选项):用 `git log` 比对自上次 GC 以来**整个仓库**新增/修改的 `.md` 文件,不要只看 `notes/` 目录——重点找游离在 `profile/concepts/decisions/pitfalls/insights/questions/notes` 七个分类目录之外的文件(用户随手扔在仓库根目录的手写/粘贴内容)。补齐分类、frontmatter、双链;一个文件混了几个不相关主题就拆分成几篇分别归位,分类或命名不对就直接改名、移动,不是只在文件末尾加标注(依据:[[decisions/整理手写笔记允许改写原文而不仅是追加复核记录]])
5. `notes/` 的 `checkable` 内容按上方「notes 的复核」流程处理;`checkable: false` 的一律跳过
6. 扫一遍本周新增的 decisions/pitfalls/insights,看有没有哪个主题下相关记忆凑够了 2 条以上、但 `SUMMARY.md` 里还没有对应条目——有就新起或更新一条(见「结论速览」一节)
6b. 概念层维护:被 ≥2 篇笔记链到的幽灵概念升格为实体页(只写定义+出现在);挑 1-2 个本周被新笔记命中的概念补延伸和开放问题;扫一眼幽灵概念列表,近义的按「同义词用 aliases 收编」规则合并(见「概念层」一节)
7. 重新生成索引并提交;向用户输出一句话摘要(动了什么、为什么)
8. 无事可做则保持沉默,不打扰用户;**不要为了排期而主动问用户"这条还作数吗"**——过时的决定/判断由用户在实际用到时当场纠正(见「三条铁律」第 3 条),GC 不做这类打扰式追问
