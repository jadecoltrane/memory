给用户设计方案(不管是这个仓库的 CLAUDE.md/GC prompt,还是笔记结构、自动化脚本),几种做法效果差不多时,优先选 token 成本更低(读起来快、维护起来轻)的那个,不要默认往最全面但最笨重的版本上靠——但省 token 不能拿效果去换,是有约束的优化,不是能省则省。

**Why:** 用户 2026-07-04 吐槽 CLAUDE.md 每次都要读好久(而且不只每次会话读,daily-workbench.yml 每天、weekly-gc.yml 每周都会整份读进去),顺带说明这是他判断任何方案的通用标准——这类工作方式偏好按 `RESOLVER.md` 的分流规则直接进 `decisions/`,不进 `notes/`(notes/ 只放文章、知识点、用户自己写的东西)。

**How to apply:** 往 CLAUDE.md、GC/工作台的 prompt、或任何会被反复读取执行的文件里加内容前,先想有没有更省 token 的等效写法;真的需要变长(比如加步骤 4a 是为了堵住自动化漏检的真实缺口)才加,不为了"讲清楚"而堆砌解释。2026-07-04 已按这个标准把 CLAUDE.md 从 175 行精简到约 140 行,后续发现新冗余可以继续精简,但不能为精简牺牲规则的可执行性。
