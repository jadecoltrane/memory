# 决定

`weekly-gc.yml` 不再做"扫描用户名下所有仓库、给缺记忆指针的仓库自动补装"这件事(原来叫"指针巡检")。以后任何仓库要接入记忆库,统一由用户在该仓库会话里说「接入记忆库」/「装上记忆」触发,AI 按 CLAUDE.md「在新仓库安装记忆指针」一节手动装,不再有后台自动扫描。

# 为什么

指针巡检依赖一个专门的 `REPO_PATROL_PAT`(需要对用户名下所有仓库有写权限的 PAT),这条链路本身维护成本不低:PAT 会过期/被误配置成范围不够(2026-07-06 就实际踩过一次,报 "Bad credentials"),排查起来还要分辨是权限范围问题还是别的上游 bug。用户本人明确表示"有新的仓库我自己加就行"——手动触发本来就是这个功能唯一的使用场景(用户自己知道什么时候建了新仓库、什么时候需要接入),自动巡检省下来的"记得手动装"这一点点操作成本,换不回维护一条额外鉴权链路的代价。

# 影响

- `weekly-gc.yml`:移除了 `REPO_PATROL_PAT` 环境变量和 prompt 里"GC 之后做指针巡检"那一段
- `CLAUDE.md`:「三个自动化」和「在新仓库安装记忆指针」两处提到巡检的地方都改成了手动触发说明
- 仓库里原有的 `REPO_PATROL_PAT` secret(以及用户排查这次问题时误建的另一个同用途 secret)不再被任何 workflow 引用,用户可以自行去 GitHub 仓库设置里删掉,留着也不影响功能,纯粹是死配置
- `templates/repo-pointer.md` 本身、以及"用户显式说接入记忆库"这条手动流程完全不受影响,继续按原来的方式工作
